# PRD — 열매똑똑 해커톤 진행상황 투명성 플랫폼 (Antigravity + Vercel)

## 0. 문서 메타
- 제품명(가칭): **열매똑똑 해커톤 보드(Hackathon Transparency Hub)**
- 버전: v0.1 (MVP)
- 작성일: 2026-02-06
- 배포 목표: **Vercel(프로덕션/프리뷰)** 기반 운영
- 개발 방식: **Antigravity(에이전트 중심 개발) + GitHub + Vercel 자동배포**

---

## 1. 문제 정의(Why)
해커톤 운영에서 자주 생기는 문제:
- 팀별 진행상황/제출물/피드백이 흩어져 있어 **운영·심사·멘토링**이 비효율적이다.
- “기술을 얼마나 만들었나(How)”가 전면에 노출되면, **과시/순위 경쟁**으로 흐르기 쉽고
  사회복지 현장에선 “왜 필요한가(Why)”가 뒤로 밀린다.
- 공개를 확대하면 투명성은 오르지만, 반대로 **위축·베끼기 논란·개인정보 사고** 위험이 커진다.

> 목표는 “팀을 이기는 플랫폼”이 아니라  
> **팀이 고객(현장) 문제를 더 빨리 검증하고 서로를 파트너로 삼게 하는 플랫폼**이다.

---

## 2. 목표(Goals)
### 2.1 제품 목표
1) 기획→프로토타입→검증→최종 결과물까지 **전 팀 진행상황을 한눈에** 보여준다.  
2) “왜 만드는가”를 강제하는 구조로 **고객(현장) 중심 사고**를 유도한다.  
3) 순위 경쟁이 아니라 **협력(피드백/도움요청/공유) 기반의 건전한 자극**을 만든다.  
4) 개인정보 사고를 막기 위해 **최소수집·비식별·접근통제**를 기본값으로 한다.

### 2.2 성공 지표(KPI) — 운영/학습 지표 중심
- 팀당 주간 **Help(도움요청)** 카드 생성률
- Help 카드 평균 **응답 리드타임(시간)**
- 팀당 **검증 로그(Experiment log)** 등록 수
- 피드백의 **유효성(“도움 됨” 클릭률)**
- 제출물 완성도(운영 기준 충족: 필수 섹션 누락률↓, 개인정보 플래그↓)

> *MVP에서는 ‘순위표/점수 공개’는 KPI에서 제외.*

---

## 3. 비목표(Non-goals)
- 실시간 협업 문서 편집(구글문서/노션 대체) — 링크로 연결
- 소스코드 호스팅(깃허브 대체) — GitHub 링크로 연결
- 과도한 커뮤니티 기능(게시판/DM) — 초기에는 피드백/멘션 중심

---

## 4. 사용자(Actors) & 권한
### 4.1 역할
- **참가자(Team Member)**: 팀 생성/참여, 팀 페이지 편집, 제출, 도움요청, 피드백 작성
- **멘토(Mentor)**: Help 목록 탐색, 피드백, “해결됨” 처리 제안
- **심사위원(Judge)**: 제출물 열람, 채점(비공개), 코멘트
- **운영자(Admin)**: 스테이지/마감/심사 배정, 신고/비식별 점검, 공개 범위 설정, 데이터 내보내기
- **대중(Public Viewer)**: 공개 범위 내 읽기 전용 열람

### 4.2 접근통제 기본값
- 심사위원 채점/코멘트는 **비공개(팀/운영자만)** 또는 운영자 설정
- 대중 공개는 **스테이지별로 점진 공개**(예: 마감 후 공개)

## 5. 핵심 사용자 경험(UX) 원칙

## 5.1 디자인 방향성(Visual + Interaction) — MVP 기준
> PRD는 “무엇/왜”를 정리하는 문서라, 픽셀 단위 시안까지 넣기보다는 **최소한의 디자인 원칙과 상호작용 규칙**을 적어
> 디자이너/개발자가 같은 방향으로 빠르게 구현할 수 있게 한다.

### A. 톤 & 메시지(‘경쟁’이 아니라 ‘학습/협력’)
- 메인 문구는 **순위/점수** 대신 **학습(배운 점)·검증(실험)·협력(도움요청/피드백)**을 전면에 배치
- 팀 카드에서 강조 순서: `스테이지 → 최근 업데이트 → Help/Insight → 기여 배지`
- 점수/석차는 **기본 숨김**(심사자/운영자 내부 화면만)

### B. 레이아웃(정보 우선, 카드 기반)
- 기본 레이아웃: 상단 공지/마감 → 필터/검색 → 팀 카드 그리드(반응형)
- 카드/리스트 혼합: 데스크톱은 그리드, 모바일은 단일 컬럼 리스트
- “필터/정렬”은 항상 화면 상단에 고정(업무용 도구는 탐색 비용이 핵심)

### C. 컴포넌트(반복을 줄이는 표준 세트)
- Team Card(상태/배지/지표), Help Card(Open/In progress/Resolved), Insight Card
- 루브릭 테이블(심사), 스테이지 배지(도입–검증–완성 등)
- 알림: 토스트(즉시), 인앱 알림(누적), 이메일(선택)

### D. 상호작용(‘왜’가 먼저 나오도록)
- 팀 페이지 첫 섹션은 항상 **Why(문제/고객/증거)** 고정(접힘 불가)
- 다음 스테이지 이동 버튼은 **필수 섹션 충족 시 활성화**
- 피드백 입력 시 템플릿(좋았던 점/걱정되는 점/확인 질문) 기본 제공

### E. 마이크로카피(쉽고 안전한 말)
- “증거” 입력란에는 개인정보 금지 문구를 짧게 고정 표기
- “베끼기” 대신 “재사용(출처+변경점)” 용어 사용
- 상태값은 쉬운 말로: Open(요청중) / In progress(진행중) / Resolved(해결됨)

### F. 접근성/가독성(필수선)
- 키보드로 이동할 때 **포커스 표시가 항상 보이게**(Focus Visible)  
  (참고: WCAG 2.2 이해자료 https://www.w3.org/WAI/WCAG22/Understanding/focus-visible.html)
- 대비/글자 크기/버튼 터치 영역을 충분히(현장 실무자 모바일 사용 고려)  
  (참고: WCAG 2.2 https://www.w3.org/TR/WCAG22/)

1) 메인 화면은 “점수/순위”가 아니라 **학습·검증 활동(Help/Insight/Experiment)** 중심
2) 팀 페이지는 **Why → 증거 → 가설 → 해결 → 실험 → 확산** 흐름을 강제
3) 공개는 “전면 공개”가 아니라 **협력 설계가 포함된 공개**
4) 개인정보는 업로드 단계에서 **체크리스트 게이트**로 차단

## 6. 핵심 화면/기능(Functional Requirements)

## 6.1 메인 대시보드(전체 한눈에)
**FR-01 팀 카드 목록**
- 팀명/기관(옵션), 현재 스테이지, 최근 업데이트(1줄), Help 개수, Insight 개수
- 필터: 스테이지, 분야(사례관리/행정/홍보/자원연계 등), “Help 있는 팀만”
- 정렬: 최근 업데이트, Help 많은 순, 응답 빠른 순(운영 목적)

**FR-02 공동 목표(협동전 지표)**
- 예: “전체가 절감한 행정시간(자기보고)” 누적 바(Bar) — 팀 경쟁이 아니라 공동 성과 강조

**FR-03 공지/마감**
- 운영자 공지(상단 고정), 마감 D-day 표시

---

## 6.2 팀 페이지(한 페이지로 기획→최종까지)
팀 페이지는 아래 섹션을 **필수**로 제공하고, 운영자가 “필수 입력”을 설정할 수 있다.

**FR-10 Why(문제/고객)**
- 대상(누구), 상황(언제), 불편(무엇) 1문장
- 증거 3개(인터뷰 메모/현장 관찰/업무 로그 등) — 개인정보 금지 안내 포함

**FR-11 가설(Hypothesis)**
- “만약 ○○를 제공하면, △△가 개선될 것이다” 형식 1~2개

**FR-12 해결(솔루션/업무흐름)**
- 업무 흐름 “전/후” 간단 도식(이미지 또는 텍스트)
- 핵심 기능 3개

**FR-13 프로토타입/데모**
- 링크(웹앱/시트/노션/앱시트 등), 사용방법 5줄
- (선택) GitHub 레포 링크, Vercel 데모 URL

**FR-14 검증 로그(Experiment log)**
- 최소 1개 실험: 대상/방법/결과/배운 점
- “가정이 틀렸던 것 1개” 항목(Insight 강제)

**FR-15 확산/운영 계획**
- 타 기관 재사용 체크리스트(필수 리소스/권한/템플릿/운영 규칙)

**FR-16 스테이지 게이팅**
- 운영자가 설정한 필수 요건 미충족 시 다음 스테이지로 이동 불가(예: Why/가설/검증로그)

---

## 6.3 업데이트/협력 기능
**FR-20 Help(도움요청) 카드**
- 팀당 주 1회 이상(운영자 옵션) 생성 유도(알림/배지)
- 형식: “지금 막힌 것 1줄 + 원하는 도움 1줄 + 관련 링크”
- 상태: Open / In progress / Resolved

**FR-21 Insight(배운 점) 카드**
- “우리가 틀렸던 가정 1개”, “다음에 검증할 것 1개” 최소 입력

**FR-22 피드백(멘토/참가자)**
- 구조화된 피드백 템플릿 제공(예: “좋았던 점 / 걱정되는 점 / 확인 질문”)
- “도움 됨” 버튼(피드백 품질 지표)
- @멘션(팀/멘토) + 알림

**FR-23 기여 배지(순위 대체)**
- 예: “피드백 5회”, “Help 해결 2회”, “템플릿 공유 1회”
- 메인 대시보드에서 팀 카드에 배지 노출(점수/순위 노출 금지)

---

## 6.4 제출/심사
**FR-30 제출 스냅샷(버전)**
- 마감 시점의 팀 페이지를 “제출 버전”으로 고정 저장
- 제출 버전에는 외부 링크(프로토타입)도 함께 기록

**FR-31 심사 배정**
- 운영자: 심사위원에게 팀 묶음 배정
- 심사위원: 할당된 팀 목록/상태/제출버전 열람

**FR-32 채점(비공개)**
- 루브릭 기반(운영자 설정): Why의 명확성, 검증의 충실성, 확산 가능성, 안전성(개인정보), 접근성 등
- 채점 결과는 기본 비공개(운영자만 집계)

**FR-33 심사 코멘트**
- 코멘트는 “공개/비공개” 토글(운영자 정책)

---

## 6.5 개인정보/신고/모더레이션
**FR-40 개인정보 체크리스트 게이트(업로드/저장 전)**
- (필수) 실명/연락처/주민번호/주소/사례 세부 식별정보 미포함 확인
- (권장) 스크린샷은 마스킹 후 업로드

**FR-41 신고/숨김**
- Public 영역에서 신고 버튼
- 운영자: 임시 숨김, 수정요청, 블라인드 처리

**FR-42 보관/삭제 정책**
- 행사 종료 후 보관 기간(운영자 설정)
- 대중 공개는 “필수 정보만” 남기고 원자료(첨부) 삭제 옵션

## 9. 기술/배포 요구사항(Antigravity + Vercel)
### 9.1 기본 기술 스택(권장)
- Front/Fullstack: **Next.js** (Vercel 친화)
- Auth: NextAuth(구글/깃허브 OAuth) 또는 이메일 매직링크
- DB: Postgres(Supabase/Neon 등) + ORM(Prisma)
- Storage: Vercel Blob 또는 Supabase Storage(첨부 제한 권장)
- Deploy: Vercel(Preview/Production)

> Vercel은 Git 저장소 연결 시 브랜치 푸시마다 자동 배포, 프로덕션 브랜치 머지 시 프로덕션 배포를 지원한다.  
> (출처: https://vercel.com/docs/git)

> Vercel Functions는 서버를 직접 운영하지 않고 서버 측 코드를 실행할 수 있는 “서버리스 함수” 형태로 제공된다.  
> (출처: https://vercel.com/docs/functions)

### 9.2 Antigravity 활용 원칙
- Antigravity는 “Artifacts(작업 산출물) + 검증 테스트” 중심 개발을 강조한다.  
  (출처: https://antigravity.google/)
- 개발 산출물(화면/기능/테스트/보안 체크리스트)을 Artifacts로 남기고, PR(풀리퀘스트) 단위로 반영한다.

### 9.3 보안 주의(에이전트 기반 개발 특성)
- 비밀키(.env), 토큰, 서비스 계정키는 **절대 저장소에 커밋 금지**  
- Antigravity/에이전트가 터미널 명령을 자동 실행할 수 있으니, “자동 실행/권한”은 최소로 두고
  배포 전 **수동 검토(코드/환경변수/권한)** 체크리스트를 운영한다.

## 10. 릴리즈 계획(제안)
### v0 (2~3주 MVP)
- 팀 등록/팀 페이지(필수 섹션)
- 메인 대시보드(필터/정렬)
- Help/Insight 카드 + 피드백 + 알림
